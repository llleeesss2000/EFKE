# EFKE 維基百科 Schema 規則

> 基於 Karpathy LLM Wiki 核心原則 + EFKE Evidence-First 理念

---

## 核心原則（Karpathy LLM Wiki）

### 1. LLM 是知識工程師，不是摘要器

> 讀 source → 提煉知識 → 組織成結構化條目 → 建立交叉引用
> 不是：複製貼上原文 → 當成維基

### 2. Schema 先行

先定義格式規則，再讓 LLM 按規則產生。沒有規則的 LLM 會自由發揮，品質不穩定。

### 3. 增量更新

新資料來時，更新現有條目，不是全部重來。維基是會累積的資產。

### 4. Lint 機制

定期讓 LLM 檢查：
- 矛盾：不同條目說法不一致
- 遺漏：重要概念沒有自己的條目
- 過時：新資料已取代舊說法
- 孤立：沒有任何其他條目引用

### 5. 人類審核

LLM 產生的內容需要人類確認。維基是知識庫，不是草稿。

## EFKE 維基改進規則

### 規則 1：主題品質篩選

不是所有出現在 chunks 中的詞都值得成為維基條目。主題必須滿足：

| 條件 | 說明 |
|------|------|
| **有實質內容** | 至少 3 個不同 chunks 提到這個概念 |
| **跨文件出現** | 至少 2 本書都提到 |
| **不是雜訊** | 不是版權頁、目錄、頁碼、電話號碼 |
| **有意義** | 是一個完整的投資概念，不是碎片 |

### 規則 2：雜訊過濾

以下內容必須被排除：

```
版權頁：作者、出版社、地址、電話、傳真、E-mail
目錄：第X章、第X節、PART X
頁碼：頁 1、page 1、P.1
UI 文字：詳見圖、如下圖所示、從圖中可以看出
廣告文字：點擊訂購、免費辦理、客服微信
重複文字：同一段落重複出現
無意義碎片：單個字、兩字詞、常見連接詞
```

### 規則 3：維基條目格式（Schema）

每個條目必須包含以下結構：

```markdown
## [主題名稱]

### 定義
（一句話精確定義這個概念是什麼）

### 核心概念
（2-4 個要點，每個要點用一句話說明）

### 運作原理/機制
（解釋這個概念如何運作，為什麼有效）

### 具體案例
（列出 evidence 中提到的所有案例，包含：
- 誰（公司名、人名）
- 做了什麼（具體操作）
- 結果如何（數字、百分比）
- 時間點）

### 計算方式（如有）
（完整寫出公式和計算步驟）

### 注意事項/風險
（列出所有提到的注意事項）

### 關聯條目
（列出與此概念相關的其他維基條目）
```

### 規則 4：LLM 合成要求

Prompt 必須明確要求：

1. **用自己的話重新組織**，不要複製貼上 evidence
2. **保留所有具體數字**（價格、張數、百分比、日期）
3. **保留所有案例**（每個案例獨立列出）
4. **每個條目至少 300 字**，上限 800 字
5. **只根據 evidence 回答**，不要補充外部知識
6. **找不到足夠 evidence 時**，不要產生條目

### 規則 5：後處理驗證

每條產生後自動檢查：

| 檢查 | 不合格標準 |
|------|-----------|
| 字數 | 少於 100 字 → 刪除 |
| 格式 | 沒有 ### 標題 → 重新格式化 |
| 雜訊 | 包含電話/地址/版權 → 刪除 |
| 重複 | 與其他條目相似度 >80% → 合併 |
| 空白 | 只有標題沒有內容 → 刪除 |

### 規則 6：交叉引用

每個條目底部必須列出「關聯條目」。規則：
- 找出 evidence 中提到的其他概念
- 用這些概念名稱作為關聯條目
- 條目之間用超連結連接

### 規則 7：來源追溯

每個條目必須標註來源：
- 哪本書（書名）
- 哪些章節/頁碼
- 使用哪個模型產生

## 完整維基產生流程

```
Step 1: 資料清理
  - 從 chunks 中過濾雜訊（版權頁、目錄、UI 文字）
  - 只保留有實質投資知識的 chunks

Step 2: 主題萃取
  - 從清理後的 chunks 中提取投資概念
  - 篩選：跨文件出現 ≥ 2、實質內容 ≥ 3 chunks
  - 去重：合併同義詞（如「本益比」=「PE Ratio」）

Step 3: Evidence 配對
  - 對每個主題，搜尋最相關的 5-8 個 chunks
  - 確保 evidence 足夠支撐一個完整的條目

Step 4: LLM 產生
  - 用 Schema 規則要求 LLM 按格式撰寫
  - 要求合成而非複製貼上
  - 要求保留所有具體數字和案例

Step 5: 後處理
  - 檢查字數（100-800 字）
  - 檢查格式（必須有 ### 標題）
  - 過濾雜訊（電話、地址、版權）
  - 合併重複條目

Step 6: 交叉引用
  - 建立條目之間的連結
  - 產生關聯條目列表

Step 7: Lint
  - 檢查矛盾
  - 檢查遺漏
  - 檢查孤立條目
```

## 實作建議

### 主題萃取改進

```python
# 不要只看頻率，要檢查實質內容
def is_meaningful_topic(topic, chunks):
    # 檢查 1：至少 3 個 chunks 提到
    if topic_count < 3:
        return False
    # 檢查 2：不是雜訊
    if is_noise(topic):
        return False
    # 檢查 3：有實質內容（不是單字、不是碎片）
    if len(topic) < 3:
        return False
    # 檢查 4：跨文件出現
    if unique_files < 2:
        return False
    return True
```

### 雜訊過濾清單

```python
NOISE_PATTERNS = [

    r'^PART\s+\d+',                      # 英文目錄
    r'作者[：:].+',                       # 版權頁
    r'出版社[：:].+',                     # 版權頁
```

### LLM Prompt 模板

```
你是「投資知識維基工程師」。你的工作是從原始證據中提煉出結構化的維基百科條目。

## 絕對規則

1. **合成而非複製**：讀懂 evidence 後，用自己的話重新組織。不要複製貼上原文。
2. **保留具體數字**：所有金額、張數、百分比、日期都必須保留。
3. **保留所有案例**：每個提到的公司、人物、操作策略都必須獨立列出。
4. **結構化輸出**：必須包含「定義 → 核心概念 → 案例 → 注意事項 → 關聯條目」。
5. **至少 300 字**：每個條目必須有足夠的內容深度。
6. **只用 evidence**：不要補充 evidence 中沒有的資訊。
7. **找不到足夠證據**：不要產生條目，直接說「證據不足」。

## 格式要求

### [主題名稱]

**定義**：（一句話）

**核心概念**：
- 要點 1
- 要點 2

**案例**：
1. [案例 1]：[誰] [做了什麼] [結果]
2. [案例 2]：...

**注意事項**：
- [注意事項 1]
- [注意事項 2]

**關聯條目**：[相關概念 1]、[相關概念 2]

## Evidence

{evidence}
```
